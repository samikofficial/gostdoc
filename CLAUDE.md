# CLAUDE.md

Инструкции для Claude Code по этому проекту. Читаются каждую сессию. Соблюдать неукоснительно.

## Назначение

Python-библиотека и CLI, которые принимают `.docx` (доклад, курсовая, ВКР, отчёт о НИР) и приводят его **оформление** к ГОСТ 7.32-2017. Меняется только оформление — не содержание.

## Главный инвариант (нарушение = баг)

Инструмент **не переписывает слова автора**. Меняются только: поля, шрифт, кегль, межстрочный интервал, отступы, выравнивание, стили абзацев, нумерация страниц, оформление подписей к рисункам/таблицам.

Слова, их порядок и орфография остаются **побайтно** прежними. Любое изменение, которое затрагивает текстовое содержание документа пользователя, — это дефект. На это есть отдельный тест (см. Definition of Done).

## Принципы разработки (обязательны)

### 1. Думать до кода
Не угадывать, не прятать непонимание, выкладывать развилки.
- Допущения называть явно. Если не уверен — спросить, а не угадать.
- Если есть несколько трактовок — показать их, а не выбрать молча.
- Если есть более простой путь — сказать об этом.
- Если запутался — назвать, что именно неясно, и остановиться.

### 2. Сначала простота
Минимум кода, решающий задачу. Ничего спекулятивного.
- Никаких возможностей сверх запрошенного.
- Никаких абстракций ради одноразового кода.
- Никакой «гибкости» и «настраиваемости», которую не просили.
- Никакой обработки невозможных сценариев.
- Если 200 строк можно уложить в 50 — переписать.
- Тест: сказал бы senior-инженер, что это переусложнено? Если да — упростить.

### 3. Хирургические правки
Трогать только необходимое. Убирать за собой только свой мусор.
- Не «улучшать» соседний код, комментарии, форматирование.
- Не рефакторить то, что не сломано.
- Держать существующий стиль, даже если сам сделал бы иначе.
- Заметил постороннюю мёртвую сущность — упомяни, но не удаляй.
- Убирай импорты/переменные/функции, которые осиротели **из-за твоих** правок; чужой мёртвый код не трогай без явной просьбы.
- Тест: каждая изменённая строка должна напрямую следовать из задачи.

### 4. Исполнение через цель
Определить критерий успеха. Крутиться в цикле, пока не проверено.
- «Добавить валидацию» → «Написать тесты на некорректный ввод, затем сделать, чтобы они прошли».
- «Починить баг» → «Написать тест, воспроизводящий баг, затем сделать, чтобы он прошёл».
- «Отрефакторить X» → «Убедиться, что тесты зелёные до и после».
- Для многошаговых задач — короткий план вида «шаг → чем проверяю».

## Технические ограничения

- Python 3.10+.
- Зависимости: `python-docx` + стандартный `re`. Больше ничего без явного обоснования в PR.
- Где `python-docx` не хватает (нумерация страниц, часть свойств секций) — спускаться в lxml через `paragraph._p` / `section._sectPr`. **Не** тащить LibreOffice, pandoc, docx2python в зависимости пакета.
- v0: **никаких** конфигов/профилей вузов. Константы ГОСТ захардкожены в одном модуле.
- v0: **никакого** ML и эвристик «на глаз» сверх описанных в ТЗ.
- Интерфейс — CLI.

## Защитное программирование (предупреждение багов заранее)

Баги в этом проекте дешевле предотвратить, чем чинить. Дисциплина:

- **Перед реализацией каждого модуля** выпиши в комментарии или в PR-описании список вероятных багов именно этого модуля и заложи защиту от них в первой же версии. Не «потом починим» — сразу.
- **Сначала тест, потом код** (TDD): пишешь падающий тест, описывающий желаемое поведение / воспроизводящий риск, затем реализацию, пока не зелёный.
- **Идемпотентность**: повторный прогон инструмента на уже обработанном файле обязан давать тот же результат. Это отдельный тест.
- **Падать громко и понятно**: на не-`.docx`, старый `.doc`, повреждённый или защищённый паролем файл — внятное сообщение с причиной и что делать, а не трассировка стека. Никаких широких `except`, глотающих ошибки.
- **Полнота обхода**: итерировать ВСЕ секции и ВСЕ абзацы, включая абзацы внутри ячеек таблиц, — но применять правила по категории абзаца, а не вслепую.
- **Не разрушать содержимое**: рисунки, формулы, OLE-объекты, гиперссылки, сноски живут внутри runs. Никогда не пересоздавай run целиком — меняй только его свойства (`rPr`). Пересоздание run теряет вложенные объекты.
- **Список известных подводных камней `python-docx`/`.docx` — в ТЗ, раздел «Известные подводные камни».** Перед кодом прочитать его целиком и заложить защиту от каждого пункта.
- Версию `python-docx` зафиксировать в `pyproject.toml`. Любой вызываемый метод API сверять через Context7, а не по памяти.

## Definition of Done

Каждая фаза закрывается автотестами (`pytest`), которые **открывают выходной `.docx`** и проверяют значения (см. критерии в ТЗ). Формулировка «работает» без проверяемого теста не принимается.

Обязательный тест инварианта: текст, извлечённый из документа до и после обработки, должен совпадать.
