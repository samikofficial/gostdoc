## 1. Контекст и цель

Студенты сдают доклады, курсовые, ВКР и отчёты в `.docx`, оформленные как попало (Calibri 11, одинарный интервал, отступы пробелами, заголовки «жирным по центру» вместо стилей). Привести оформление к ГОСТ 7.32-2017 вручную — рутина на часы.

**Цель v0:** CLI-инструмент, который принимает `.docx` и возвращает `.docx` с оформлением по ГОСТ 7.32-2017 (поля, шрифт, интервал, абзацный отступ, выравнивание, нумерация страниц, оформление существующих стилей заголовков и подписей), **не меняя ни одного слова исходного текста**.

Чего v0 сознательно **не** делает (см. раздел 6): не распознаёт неразмеченные заголовки, не правит список литературы, не учитывает методички конкретных вузов.

## 2. Принципы разработки (обязательны, дублируют CLAUDE.md)

### 1. Думать до кода
Не угадывать. Не прятать непонимание. Выкладывать развилки.
- Допущения называть явно. Если не уверен — спросить, а не угадать.
- Если есть несколько трактовок — показать их, а не выбрать молча.
- Если есть более простой путь — сказать об этом.
- Если запутался — назвать, что неясно, и остановиться.

### 2. Сначала простота
Минимум кода, решающий задачу. Ничего спекулятивного.
- Никаких возможностей сверх запрошенного.
- Никаких абстракций ради одноразового кода.
- Никакой «гибкости»/«настраиваемости», которую не просили.
- Никакой обработки невозможных сценариев.
- Если 200 строк можно уложить в 50 — переписать.
- Тест: сказал бы senior-инженер, что это переусложнено? Если да — упростить.

### 3. Хирургические правки
Трогать только необходимое. Убирать только свой мусор.
- Не «улучшать» соседний код, комментарии, форматирование.
- Не рефакторить то, что не сломано.
- Держать существующий стиль, даже если сделал бы иначе.
- Заметил посторонний мёртвый код — упомяни, не удаляй.
- Убирай импорты/переменные/функции, осиротевшие из-за твоих правок; чужой мёртвый код не трогай без просьбы.
- Тест: каждая изменённая строка прямо следует из задачи.

### 4. Исполнение через цель
Определить критерий успеха. Крутиться в цикле, пока не проверено.
- «Добавить валидацию» → «Написать тесты на некорректный ввод, затем сделать, чтобы прошли».
- «Починить баг» → «Написать тест, воспроизводящий баг, затем сделать, чтобы прошёл».
- «Отрефакторить X» → «Убедиться, что тесты зелёные до и после».
- Многошаговые задачи — план вида «шаг → чем проверяю».

## 3. Главный инвариант продукта (нарушение = дефект)

Инструмент меняет **только оформление**: поля, шрифт, кегль, цвет, межстрочный интервал, интервалы до/после абзаца, абзацный отступ, выравнивание, нумерацию страниц, оформление существующих стилей заголовков и подписей к рисункам/таблицам.

**Текстовое содержание тела документа — слова, их порядок, орфография — остаётся побайтно прежним.** Единственный текст, который инструмент вправе добавить, — поле номера страницы в колонтитуле.

Проверяется тестом `test_invariant` (раздел 10): текст тела до и после совпадает.

## 5. Технологический стек и ограничения

- **Python 3.10+**.
- **Зависимости:** `python-docx` (зафиксировать версию в `pyproject.toml`) + стандартный `re`. Тесты: `pytest`. Больше ничего без явного обоснования.
- Где `python-docx` не хватает (нумерация страниц, тонкие свойства секций, слоты шрифтов) — спускаться в lxml через `._element` / `._p` / `._sectPr`. **Не** добавлять в зависимости LibreOffice, pandoc, docx2python, lxml сверх того, что тянет сам `python-docx`.
- Интерфейс — CLI. Без GUI, без веба, без сети.
- Никаких конфигов/профилей вузов в v0. Все ГОСТ-значения — в одном модуле `constants.py`.

> **Заметка по уверенности (принцип №1):** значения полей, абзацного отступа, правила заголовков, нумерации, подписей — взяты прямо из текста стандарта. Кегль 14 и интервал 1,5 — устойчивая практика оформления ВКР/курсовых по 7.32-2017 (сам стандарт задаёт «кегль не менее 12» и единую гарнитуру). Кегль таблиц/подписей в разных методичках бывает 12 — в v0 оставляем 14 единообразно и помечаем как кандидата на профили (Фаза с конфигами). Менять эти числа — только правкой `constants.py`, нигде не хардкодить повторно.

### 7.2 Категории абзацев и правила (v0 работает по стилям)

Каждый абзац относим к одной категории по его **стилю** (не по виду). Правила:

- **body** (стиль `Normal`/`Обычный`/`Body Text`, не в таблице, не список): TNR 14, интервал 1,5, до/после 0, отступ первой строки 1,25 см, по ширине, цвет авто.
- **heading** (стиль начинается с `Heading`/`Заголовок`): TNR 14, полужирный, без подчёркивания; интервал и отступ — по правилам заголовка; цвет авто. (Преобразование «жирного по центру» в заголовок — Фаза 2, не здесь.)
- **caption** (стиль `Caption`/`Подпись`): TNR, без абзацного отступа, одинарный интервал; выравнивание не навязываем агрессивно.
- **table-cell** (любой абзац внутри ячейки): TNR, одинарный интервал, **без** абзацного отступа первой строки; выравнивание не трогаем.
- **list-item** (есть `numPr`): TNR 14, интервал 1,5; **первую строку не сдвигаем** (отступ списка задаётся нумерацией) — body-отступ не применять.
- **toc / прочее служебное**: не трогать содержимое и порядок; допускается только смена гарнитуры/кегля.
- **empty** (пустой абзац): не добавлять отступ/выравнивание, чтобы не плодить артефакты.

Если стиль абзаца неизвестен — относим к body **только** если абзац вне таблицы, без `numPr` и непустой; иначе оставляем как есть (консервативно).

### 7.4 CLI

```
gostdoc INPUT.docx [-o OUTPUT.docx] [--check]
```
- Без `-o` — результат пишется рядом: `INPUT.gost.docx` (не перезаписывать исходник по умолчанию).
- `--check` — ничего не пишет; открывает документ и проверяет соответствие ГОСТ-константам, печатает список несоответствий, код выхода `0/1`. (Этот же код используется в тестах и помогает циклу «цель → проверка».)
- Понятные сообщения об ошибках (раздел 8, пункты 12–13), код выхода ≠ 0.

### 7.5 Алгоритм конвейера (`formatter.py`)

```
1. Открыть документ (обработать ошибки: не .docx / .doc / повреждён / запаролен).
2. styles.py:    выставить docDefaults и стиль Normal (TNR 14, 1.5, отступ, по ширине, цвет авто);
                 привести стили Heading*/Caption к ГОСТ.
3. sections.py:  для КАЖДОЙ секции выставить поля; настроить нумерацию страниц
                 (different first page + поле PAGE по центру в основном колонтитуле).
4. Обход тела: для каждого абзаца (включая абзацы в ячейках таблиц и в колонтитулах тела):
     a. classify.py -> категория;
     b. paragraphs.py -> интервал/отступ/выравнивание по категории;
     c. для каждого run в абзаце: runs.py -> гарнитура/кегль/цвет, сохранив b/i/u/vertAlign;
        runs, содержащие <w:drawing>/<w:object>/поле, не пересоздавать — менять только rPr.
5. Сохранить в OUTPUT.
6. (Самопроверка) при флаге --check — прогнать проверки соответствия.
```

Каждый шаг идемпотентен: повторный прогон не меняет уже корректный документ.

## 8. Известные подводные камни `python-docx`/`.docx` — предотвратить ЗАРАНЕЕ

Это перечень будущих багов. По каждому — обязательная мера в первой версии кода.

1. **Прямое форматирование перебивает стиль.** Смена шрифта в стиле `Normal` НЕ меняет run'ы с прямым форматированием (студент вручную задал Calibri 11 на самом тексте). → Нормализовать гарнитуру/кегль/цвет **на уровне run'ов**, а не только в стиле. Стиль задаём как базу, run'ы — приводим явно.

2. **Сохранение смысловых акцентов.** При нормализации run'ов **сохранять** `b` (полужирный), `i` (курсив), `u` (подчёркивание), `vertAlign` (над/подстрочный). Приводить принудительно только гарнитуру, кегль, цвет. Иначе потеряются формулы-индексы и выделения.

3. **Несколько секций.** Документ может иметь много секций, у каждой свои поля и колонтитулы. → Итерировать `document.sections` целиком, а не `sections[0]`. Классический баг «починилась только первая страница».

4. **Номера страниц — нет нативного API.** `python-docx` не умеет поле номера штатно. → Вставлять поле через XML (`w:fldChar begin` → `w:instrText` = `PAGE` → `w:fldChar end`) в абзац колонтитула, выровненный по центру. Хелпер в `xml_utils.py`.

5. **Титульный лист без номера, но в сквозной нумерации.** → На первой секции включить `different_first_page_header_footer = True`, первый-страничный колонтитул оставить пустым, поле `PAGE` положить в основной колонтитул. Нумерация при этом считает титул страницей 1, но не печатает на нём номер — ровно как требует ГОСТ. Не сбрасывать старт нумерации.

6. **Слоты шрифта `w:rFonts`.** Установка только `ascii` может дать фолбэк на кириллице. → В хелпере ставить `w:ascii`, `w:hAnsi`, `w:cs` (а при наличии `w:eastAsia` — и его) одним значением.

7. **`docDefaults` важнее, чем кажется.** Истинные дефолты документа лежат в `styles.xml` → `w:docDefaults`. Правка только стиля `Normal` иногда не покрывает всё. → Выставлять и `docDefaults` (rPr/pPr), и `Normal`.

8. **Локализованные имена стилей.** В русском Word — `Обычный`, `Заголовок 1` вместо `Normal`, `Heading 1`. Доступ по имени может не найти стиль. → Работать по `style_id` где возможно; держать карту синонимов имён; отсутствующий нужный стиль — создавать, а не падать.

9. **Правило межстрочного интервала.** Если у абзаца `line_spacing_rule = EXACTLY` с точками, простое присваивание `1.5` ведёт себя иначе. → Явно ставить правило `MULTIPLE` и `line_spacing = 1.5`.

10. **Абзацный отступ не везде.** `first_line_indent = Cm(1.25)`, применённый ко ВСЕМ абзацам, ломает списки, подписи, оглавление. → Применять только к категории **body** (раздел 7.2). Спискам/подписям/таблицам/служебным — не применять.

11. **Абзацы внутри таблиц.** Текст ячеек — тоже абзацы. Слепое применение body-правил коверкает таблицы (отступ первой строки, выравнивание по ширине). → Категория **table-cell**: TNR, одинарный интервал, без отступа первой строки, выравнивание не трогаем.

12. **Не-`.docx` и старый `.doc`.** `python-docx` не открывает бинарный `.doc` и кинет невнятную ошибку. → Проверять расширение/сигнатуру; на `.doc` — сообщение «Формат .doc не поддерживается, пересохраните в .docx». На прочее — «Это не файл .docx».

13. **Повреждённый/запароленный файл.** → Ловить ошибку открытия, отдавать понятное сообщение, код выхода ≠ 0. Без трассировки стека наружу. Никаких широких `except`, глотающих всё подряд.

14. **Идемпотентность.** Повторный прогон не должен «удваивать» эффекты (второй пустой колонтитул, второе поле PAGE и т.п.). → Перед вставкой поля PAGE проверять, что его ещё нет. Тест `test_idempotency`: `process(process(x)) == process(x)` по проверяемым свойствам.

15. **Инвариант текста vs добавленный номер.** Тест «текст до == после» сломается из-за поля PAGE в колонтитуле. → Сравнивать текст **тела** (абзацы документа и ячейки таблиц), исключая колонтитулы и поля. Хелпер извлечения «чистого тела» — общий для теста и кода.

16. **Рисунки/объекты в run'ах.** Пересоздание run теряет `w:drawing`/`w:object`. → Никогда не создавать новый run взамен старого; менять только `rPr` существующего.

17. **Сноски/гиперссылки.** Лежат в отдельных частях и в run'ах. → Не трогать их содержимое; гарнитуру в сносках приводить тем же хелпером, но не переписывать текст.

18. **Зеркальные поля / переплёт (`mirrorMargins`, gutter).** Левое/правое могут меняться местами. → Если включены зеркальные поля — нормализовать к обычным левому/правому по ГОСТ.

19. **Tracked changes / комментарии.** Если в документе есть правки/комментарии, грубое вмешательство в XML может их побить. → v0: не пытаться их «принять/отклонить», но и не ломать. Если ревизии присутствуют — предупредить в выводе, что итог может выглядеть иначе после их принятия.

20. **Версия `python-docx`.** Поведение API меняется между версиями. → Зафиксировать версию; неочевидные вызовы сверять через Context7.

21. **Пустой/вырожденный документ** (нет секций, только картинка, гигантский файл). → Не падать; пустые абзацы пропускать; на больших файлах не держать лишних копий в памяти.

## 10. Definition of Done (критерии приёмки v0)

v0 закрыт, когда `pytest` зелёный и каждый пункт ниже покрыт тестом, **открывающим выходной `.docx`**:

- **Инвариант:** `test_invariant` — чистый текст тела входа и выхода идентичен (раздел 8.15).
- **Поля:** для **каждой** секции `left=Mm(30)`, `right=Mm(15)`, `top=Mm(20)`, `bottom=Mm(20)`.
- **Шрифт:** `docDefaults` и стиль `Normal` → Times New Roman; run'ы тела → Times New Roman во всех слотах `rFonts`; кегль body = `Pt(14)`.
- **Сохранение акцентов:** run, который во входе был полужирным/курсивом/над-подстрочным, остаётся таким на выходе.
- **Интервал/отступ:** body → интервал 1,5, до/после 0, отступ первой строки `Cm(1.25)`, выравнивание по ширине; абзацы в таблицах — без отступа первой строки и с одинарным интервалом.
- **Нумерация:** в основном колонтитуле есть поле `PAGE`, выравнивание по центру; на первой странице номер не отображается; нумерация сквозная.
- **Стили заголовков:** существующие `Heading*` → полужирный TNR без подчёркивания.
- **Идемпотентность:** `test_idempotency` — повторный прогон не меняет проверяемых свойств и не добавляет второго поля PAGE/пустого колонтитула.
- **Ошибки:** `test_errors` — на `.doc`, не-`.docx`, повреждённый файл выдаётся понятное сообщение и код выхода ≠ 0, без трассировки.

«Работает» без зелёного теста на пункт — не принимается (принцип №4).
